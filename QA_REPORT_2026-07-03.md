## Light Theme — Fixed

### Root-cause bugs (highest impact)
1. **`.btn-close` (modal X button) was inverting to white unconditionally**, regardless of theme — made every modal's close button invisible on light theme's near-white modal headers. Fixed: inversion now scoped to `[data-theme="dark"]` only. This one fix resolved the same bug across 5 modal usages app-wide (watchlist, portfolio, markets index, admin API configs).
2. **Legacy CSS variables `--card-bg`, `--input-bg`, `--surface-2` were never overridden inside `[data-theme="light"]`** — any component still using these older variable names stayed permanently dark-themed even after switching to light mode. Fixed by re-pointing them to the correct light-theme tokens inside the light theme block.
3. **Top navbar had a hardcoded dark `rgba(13,17,23,0.92)` background** regardless of theme — fixed to use theme-aware tokens.
4. Removed a redundant/duplicate `.btn-close` override rule left over from an earlier fix pass (dead CSS, no functional impact, cleaned up for maintainability).

### Component-level contrast fixes
- Translucent borders/backgrounds (`.confidence-bar`, `.progress`, `.badge-tag`, `.kpi-icon`, `.bg-glass`) using low-opacity white (`rgba(255,255,255,0.0x)`) — nearly invisible on white backgrounds. Fixed with theme-appropriate values.
- Date `<input>` fields had hardcoded `color-scheme: dark`, forcing a dark native picker UI even in light theme.
- Chart.js canvases in **model_performance.html**, **backtesting.html**, **performance.html** had hardcoded gray text/gridline colors (`#94a3b8`, `#64748b`, `rgba(255,255,255,0.04)`) baked into chart configs — fixed to read the live CSS variable via `getComputedStyle()` at render time, so charts now render correctly in both themes.
- **advanced_analysis.html**: Fibonacci table price cells and liquidity pool price spans had hardcoded `color:#fff` (invisible on light backgrounds) — fixed to `var(--text-primary)`.
- **asset/detail.html**: signal-history timestamp cell hardcoded `color:#fff`; `.tab-nav .nav-link.active` hardcoded white; RSI neutral-state color hardcoded white — all fixed to theme-aware tokens.
- **dashboard/watchlist.html**: confluence badge always rendered white text even for the "neutral" case where the background falls back to a near-transparent border color, causing white-on-white — fixed with a conditional (white only when the badge has a solid green/red fill).
- **dashboard/auto_generate.html**: timestamp and confidence-label text hardcoded light-gray hex colors — fixed to `var(--text-muted)`.
- **markets/commodities.html**, **markets/index.html**: hardcoded muted-gray text and a translucent-white regenerate-button background — fixed.
- **dashboard/ta_summary.html**, **dashboard/mtf_analysis.html**: progress-bar/mini-bar track backgrounds using low-opacity white — fixed to `var(--border)`/`var(--bg-secondary)`.
- **heatmap.html**, **model_performance.html**: additional invisible translucent borders/hover states — fixed.

### Reviewed — confirmed correct, no change needed
- All "white text" instances sitting on a **solid colored background** (accent purple buttons, green/red P&L bars, tab-active states, gradient Start/Stop buttons) — contrast holds in both themes by design, left as-is.
- **advanced_analysis.html** price chart (`#aaChartWrap`) — intentionally kept permanently dark (documented in-file comment: matches TradingView/Bloomberg convention so candle colors stay legible regardless of app theme).
- **partials/base.html sidebar** — intentionally a self-contained dark strip in both themes (common enterprise dashboard pattern), internally consistent, not a contrast bug.
- **landing.html** — standalone marketing/login page with no theme toggle exposed to the user; out of scope for this pass.
- briefing.html, economic_calendar.html — checked, already theme-aware from initial build.

**Verification**: CSS brace-balance checked after every edit (stayed valid throughout), and all previously-working page routes (34 total + 5 market sub-pages) re-tested as HTTP 200 after the full set of changes was applied and the server restarted. No backend/Python files were touched during this pass.

## Functional QA — Pages & API

### Page load health (all 34 routes + 5 market variants)
All return HTTP 200. No Jinja/Python tracebacks leaking into any rendered page (scanned every page body for `jinja2.exceptions`, `UndefinedError`, `TemplateNotFound`, `Traceback` — zero hits).

### Authenticated API spot-checks (via Flask test client, no real credentials used)
- `GET/PUT /auth/me` (Risk Manager's Account Size / Risk % save) — **works correctly**, persists and reloads properly.
- `POST /scanner/run`, `GET /scanner/filters` — working, returns clean empty-result set when no matches (not an error).
- `POST /portfolio/add` with an invalid symbol — correctly returns `404 Asset not found`, not a crash.
- `GET /assets/search?q=...` (Delta-first admin search) — working, Delta results correctly tagged and returned first.
- `POST /assets/add-from-search` on an asset that already exists — correctly returns `409` with the existing asset's data (the bug fixed earlier this session is confirmed still working after all later changes).
- `GET /trading/status|balances|positions|orders` with no Delta credentials configured — all correctly return `503` with a clear actionable message ("Add your real credentials in Admin → API Configs"), never a raw crash. Confirmed safe with zero real trading credentials in the system.
- Malformed/empty request bodies (`portfolio/add`, `scanner/run`) — correctly return `400`, not `500`.
- `journal/`, `notifications/`, `predictions/model-performance` — all load without error.
- Auto Generate: `status`, `watchlist`, `save`, `start` endpoints all confirmed working; the DB-backed config (built earlier this session) persisted correctly across a server restart mid-session (`3 assets × ['5m','15m','30m'] every 5min` resumed as expected).

### Minor finding (not fixed — flagged for your review)
- **`POST /risk/position-size` and `POST /risk/risk-reward`** are real backend endpoints (`app/api/v1/risk.py`) but are **not called by any frontend page** — the Risk Manager's Trade Calculator does all position-sizing math client-side in JavaScript instead. These two endpoints appear to be dead/orphaned code (possibly built for a feature that was later reimplemented client-side, or intended for future reuse — e.g. a mobile app or public API). Not broken, just unused. Recommend deciding whether to: (a) leave them as a reusable calculation API, (b) wire the frontend to use them instead of duplicating the math client-side, or (c) remove them if truly dead. No action taken — flagging for your call.
- The backend error for malformed risk-calculator input returns a raw Python `KeyError` string (e.g. `"'capital'"`) instead of a friendly validation message. Low priority since the endpoint isn't currently used by any UI, but worth cleaning up if you decide to wire it up per the point above.
